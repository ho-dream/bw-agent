# Engineering: Harnessing Codex in an Agent-First World

> Source: https://openai.com/index/harness-engineering/
> Author: Ryan Lopopolo, OpenAI
> Date: February 11, 2026

## 概述

OpenAI 团队进行了一项实验：构建并交付一款软件产品的内部 beta 版，其中没有一行代码是人工编写的。每一行代码（应用逻辑、测试、CI 配置、文档、可观察性、内部工具）全部由 Codex 编写。估计只用了手工编写代码所需约 1/10 的时间。

**人类掌舵。智能体执行。**

团队用几周时间交付了最终达到一百万行代码的项目。~1,500 个 PR 被合并，由三名工程师推动（平均每人每天 3.5 个 PR）。

## 从空的 Git 仓库开始

初始架构（仓库结构、CI 配置、格式化规则、包管理器、应用框架）由 Codex CLI 使用 GPT-5 生成。甚至 AGENTS.md 文件也由 Codex 编写。

## 重新定义工程师的角色

工程师工作重点转向系统、架构和杠杆作用。核心问题从"写什么代码"变为"究竟还需要什么样的能力，如何让这个能力对智能体来说既清晰可读又可强制执行？"

人类通过提示与系统交互：描述任务 → 运行智能体 → 允许其打开 PR。使用 Ralph Wiggum 循环：Codex 审核自身更改 → 请求智能体审查 → 响应反馈 → 循环直到所有审核人员满意。

## 提高应用程序的可读性

- 应用可根据 git worktree 启动，Codex 可为每次更改启动独立实例
- Chrome DevTools 协议接入智能体运行时，支持 DOM 快照、截图和导航
- 本地可观测性堆栈（LogQL 查询日志，PromQL 查询指标）
- 单次 Codex 运行可持续 6+ 小时

## 代码仓库设为记录系统

### 失败尝试：大型 AGENTS.md

- 上下文是稀缺资源，巨大指令文件会挤掉任务和代码
- 过多指导变得无效，智能体进行模式匹配而非有意识导航
- 立即腐烂，难以核实

### 成功方案：AGENTS.md 作为目录

AGENTS.md（~100 行）仅作为地图，指向 docs/ 中的深层信息：

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

实现**渐进式披露**：智能体从小而稳定的切入点开始，被指导下一步去哪里查看。

专职 linter 和 CI 作业验证知识库的更新状况。定期运行 "doc-gardening" 智能体扫描过时文档。

## 智能体可读性优先

- 智能体运行时无法访问的内容等于不存在
- Slack 讨论、Google Docs 中的知识必须推送到仓库
- 倾向于选择可完全内化于仓库推理的依赖项
- "枯燥"技术更易被智能体建模（可组合性、API 稳定性、训练集表现）

## 规范架构与品味

通过强制执行不变量（而非微观管理实施）来保持连贯性：

- 严格的架构模型：每个业务域划分固定层（Types → Config → Repo → Service → Runtime → UI）
- 横切关注点通过单一 Providers 接口进入
- 自定义 linter 和结构测试机械地强制执行
- lint 错误信息中注入修复指令（正向 prompt injection）

**品味不变式**：结构化日志、命名约定、文件大小限制、平台可靠性要求。

> 在以人为本的工作流中，这些规则可能让人感到束缚。有了智能体，它们就成了倍增器：一旦编码，立即应用于所有地方。

## 吞吐量改变了合并理念

- PR 生命周期很短
- 测试偶发失败通过后续重跑解决
- 纠错成本低，等待成本高

## "智能体生成"的完整范围

智能体产出包括：产品代码与测试、CI 配置和发布工具、内部开发者工具、文档和设计历史、评估框架、审阅评论和回复、管理仓库的脚本、生产仪表板定义。

## 自主水平提升

给定提示，智能体现可端到端地：

1. 验证代码库当前状态
2. 重现已报告的 Bug
3. 录制演示故障的视频
4. 实施修复
5. 运行应用验证修复
6. 录制第二个演示视频
7. 打开 PR
8. 响应智能体和人类反馈
9. 检测并修复构建故障
10. 仅在需要判断时交由人工处理
11. 合并更改

## 熵与垃圾收集

Codex 会复现仓库中已有模式（包括不够理想的模式），导致漂移。

### 解决方案

将"黄金原则"编码到仓库中，建立循环清理流程：

- 定期运行后台 Codex 任务扫描偏差
- 更新质量等级
- 发起有针对性的重构 PR
- 大多数可在一分钟内审查并自动合并

> 功能类似于垃圾回收。技术债务就像高息贷款：小额持续偿还总比累积后痛苦一次解决要好。

## 仍在学习的内容

> 我们当前最棘手的挑战集中在设计环境、反馈回路和控制系统方面。

开放问题：
- 架构连贯性如何随时间演变
- 人类判断力在哪里发挥最大作用
- 如何编码人类判断力使其发挥更大作用
- 随着模型能力增强，系统如何演变
